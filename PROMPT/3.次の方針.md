次の研究方針のアドバイスです。

---

## 結論

採るべき方針は、単純な A または B ではなく、**B → C₀ → A** です。

1. **現在地点を v0.2 として直ちに固定する**
2. **最終合成定理の骨格だけを先に Lean で完成させる**
3. **`cpow` 範囲縮約を既存経路と並存する v2 として実装する**
4. **小規模な性能・健全性ゲートを通過した後だけ、ブロック26以降を再開する**

現在のキャンペーンは再起動しないでください。`cpow` 実装または対応補題のハッシュが変わるまでは、同じ失敗を再投入しない状態にすべきです。

公開履歴でも、2026年7月22日の `69c3ae4` は「ブロック0–25完了、26/28は高t `cpow` 律速、角度範囲縮約が必須」と明確に記録されています。ブロック26では一部セルが既に kernel-checked である一方、後続セルが600秒でタイムアウトしています。 ([GitHub][1])

## なぜ「Bを先に行い、そのままAへ進む」のか

今回の停止は、数理戦略が崩れた兆候ではありません。ログ上では、タイムアウトしたセルでも `margin≈1.06–1.19` とかなり余裕があり、数値的不成立よりも生成証明のサイズ・elaboration・`norm_num` 処理量が支配的である可能性が高いです。ただし、これらのログはあくまで未信頼診断情報であり、数学的事実として扱ってはいけません。 ([GitHub][1])

つまり、現時点の成果を固定する価値と、`cpow` を改修する必要性は両立します。

また、現在公開されているタグは、第一零点存在を end-to-end で kernel-check した `v0.1-autonomous-math-benchmark` のみです。今回の大量被覆、経路Cの部品、キャンペーン堅牢化は、別の再現可能なマイルストーンとして切るべき量に達しています。 ([GitHub][2])

## Phase 0: v0.2 を固定する

`69c3ae4` を計算スナップショットとして参照し、その直後に**文書だけのコミット**を置いて、annotated tag を打つのがよいです。

推奨名は、たとえば次です。

```text
v0.2-t13-offline-certificate-milestone
```

ここで重要なのは、v0.2 の説明を過大にしないことです。

最終的な Z-A/Z-B 合成定理がまだないなら、

> t≤13 に線外零点が存在しないことを証明した

とは書かず、

> Z-A off-line のブロック0–25について、セルレベル証明書の生成・検証・昇格が完了した

と書くべきです。

v0.2 文書には、成果を少なくとも次の4区分で記録します。

* **KernelChecked / promoted**
* **KernelChecked / unpromoted**
* **数値実証または構造検証のみ**
* **未完了の合成・証明書生成**

さらに、環境digest、対象コミット、セル数、claim ID一覧、再生コマンド、監査結果、既知の未完了点を固定します。リポジトリ自身が「LLM・Rust・Prolog・ログではなく、ピン留めされたLeanカーネルと公理監査だけを根拠にする」と定義しているため、この区分は研究上の説明ではなく信頼モデル上の必須事項です。 ([GitHub][3])

## Phase C₀: 計算再開前に「最終合成の骨格」を完成させる

ここが、Aへ直行する前に追加すべき最重要工程です。

残りの証明書を仮定として受け取り、最終目標を導くLean定理を先に作ります。概念的には次の形です。

```lean
theorem endgame_of_certificates
    (hZA_0_25 : ZAOfflineCoverage0To25)
    (hZA_26   : ZAOfflineCoverage26)
    (hZA_28   : ZAOfflineCoverage28)
    (hDisk    : FirstZeroDiskExclusion)
    (hXiPrime : XiPrimeBoundaryCertificate)
    (hZB      : ZBLineCertificate) :
    ExactTargetStatement := by
  ...
```

これは `sorry` や新しい公理を必要としません。未完成の証明書を通常の仮定として受け取れば、**合成論理そのものは今すぐkernel-checkできます**。

この段階で確定させるべきものは、次です。

* ブロック間の端点が閉区間・開区間のどちらで接続されるか
* ディスク列と長方形被覆の間に空隙がないか
* Z-A と Z-B が同一の零点概念・重複度・領域定義を使っているか
* λ₃による除数非零条件が、実際に除算を行う全地点を覆っているか
* Ξ′証明書が最終合成で必要な形と完全に一致しているか
* 「第1零点存在」と「それ以前の線外零点排除」から何を正確に導くのか
* 最終定理がリーマン予想全体なのか、第一零点までの局所的結論なのか

この合成定理が既に存在する場合も、残る各leaf claimとの型レベル対応を監査してください。「全経路をマッピング済み」と「最終定理が証明書を仮定としてkernel-check済み」は別物です。

長時間計算の前にこれを行うことで、最後に量化子、境界、零点の重複度、除算条件などの不一致が発覚する危険を消せます。リポジトリのsecurity modelも、カーネルが「形式化された文」と「意図した数学」の乖離までは検出できないことを残余リスクとして明記しています。 ([GitHub][4])

## Phase A: `cpow` 範囲縮約の正しい実装

提案されている方針は正しいですが、次の形に限定すべきです。

角度 (\theta) について

[
|\theta-d|\le r,
\qquad
|q-2\pi|\le\varepsilon,
\qquad
k\in\mathbb Z
]

があるとします。縮約中心を

[
d_{\mathrm{red}}=d-kq
]

とすると、

[
|(\theta-2\pi k)-d_{\mathrm{red}}|
\le r+|k|\varepsilon
]

です。そして周期性は近似値ではなく、**正確な (2\pi)** に対して使います。

[
\sin\theta=\sin(\theta-2\pi k),\qquad
\cos\theta=\cos(\theta-2\pi k)
]

ここで絶対にしてはいけないのは、

[
\sin\theta=\sin(\theta-kq)
]

のように、有理近似 (q) 自体を周期として扱うことです。(q) は縮約後の球中心を有理数にするためだけに使い、その差は必ず半径に入れます。

### 355/113について

`355/113` は (\pi) の近似なので、(2\pi) の中心として使う値は

[
q=\frac{710}{113}
]

です。

実際、

[
\left|\frac{710}{113}-2\pi\right|
\approx 5.3353\times10^{-7}
]

です。たとえばLean側で

[
\left|\frac{710}{113}-2\pi\right|\le 6\times10^{-7}
]

を証明できれば、(|k|\le8) でも追加半径は最大 (4.8\times10^{-6}) です。最終セルのログ上の余裕に比べれば小さそうですが、判定には最終セルmarginではなく、**各 `cpow` trig ball の局所半径予算**を使ってください。

すでにπ因子用のkernel-checkedなπ球があるなら、新たなπ評価機構を作らず、そこから

[
2\pi\in \operatorname{Ball}(2p,2r_\pi)
]

を導出して共通利用するのがよいです。

### 実装境界

範囲縮約は `Complex.cpow` 全体や複素対数に対して適用せず、正の実数底について既に証明された分解後の、**実数 `sin` / `cos` 位相部分だけ**に適用してください。これによりbranch cutを議論に持ち込まずに済みます。

Lean側には、再利用可能な1つの補題を用意します。

```text
trig_ball_reduce_two_pi
```

概念的な入力は、元角度球、整数 `k`、(2\pi) の有理球です。出力は、縮約された角度球と元のsin/cosとの同一性です。

この補題の証明を各セルにインライン展開してはいけません。名前付き共通定理として一度だけコンパイルし、各 `cpow` 証明では定理適用と有理数計算だけを生成します。

## 既存経路を置き換えず、v2を並存させる

既存の `trig_chain` を直接書き換えるのは避け、概念上は次の戦略型にします。

```rust
enum CpowTrigStrategy {
    LegacyDoubling,
    PeriodicReductionV2 { k: BigInt },
}
```

`q`、(\varepsilon)、縮約中心、縮約半径をすべて証明書データとして重複保持する必要はありません。固定された共有定理から導出できるものは導出し、各証明書に入れる未信頼データは原則として整数 `k` だけにします。

`k` は浮動小数点で選ばず、元の有理中心 `d` と有理周期中心 `q` から、決定的な最近接整数として選びます。正負の角度を扱うため `Nat` ではなく `Int` にしてください。

既存v1とv2の結論となるLean命題は同一に保ちます。そうすれば、

* ブロック0–25の既存証明は再生成しない
* ブロック26で既に成功したセルも保持する
* v2は失敗・未完了セルだけに使用する
* 最終合成ではv1とv2の証明書を混在させる

という運用ができます。

## 実装後のゲート

フルキャンペーンを起動する前に、次の順で通してください。

### 1. 補題単体

* (k=0)
* 正の `k`
* 負の `k`
* 縮約中心が (\pm\pi) 付近
* 入力球が縮約境界をまたぐ場合
* 大きな `k`

境界をまたいでも健全性には問題がない設計にし、違いは性能だけに限定します。

### 2. 誤り注入

次の改変がLeanに拒否されることを確認します。

* (|k|\varepsilon) を半径から削除
* 縮約中心の符号を反転
* `d-kq` を `d+kq` に変更
* 証明済みより小さい (\varepsilon) を使用
* exact (2\pi k) ではなく (kq) を周期として使用

このテストは通常のunit test以上に重要です。生成器が誤ったときに、Leanが確実に拒否することを確認するテストだからです。

### 3. 低tでの差分試験

ブロック0–25から代表点を取り、v1とv2を両方kernel-checkします。新しい球が既存球と合理的に重なるか、半径が異常に膨張していないかを比較します。

ただし、全ブロックを再計算する必要はありません。代表点だけで十分です。

### 4. 高tパイロット

次の順序がよいです。

1. (d_0\approx47) の最悪 `cpow` 点を1個
2. ブロック26の未完了セルを1個
3. チェーングループを1個
4. ディスク列用 `cpow` を1個
5. Ξ′用 `cpow` を1個

ブロック26全体を先に完了してからディスクやΞ′を試すのではなく、**異なる利用経路を1点ずつ早期に通す**べきです。そうすれば、被覆用には適合するがΓ/Ξ′経路では使えない、という設計ミスを早く検出できます。

### 5. 性能基準

推奨するデフォルト基準は次です。

* 代表セルのP95検証時間が現在の600秒timeoutの20%以下
* 最悪 `cpow` 点で少なくとも5倍、できれば10倍の高速化
* 生成Leanソースまたは証明ステップ数が旧経路の30%以下
* 追加半径が既存trig ball半径の10%以下、または最小局所余裕の20%以下
* verifier、audit allowlist、公理集合を変更していない
* 誤り注入テストがすべて拒否される

これを満たさない状態でブロック28まで投入するべきではありません。

2π縮約だけでまだ遅い場合は、次の段階として正確な象限恒等式を使い、角度を ([-\pi/4,\pi/4]) に落とします。いきなり象限縮約まで実装せず、まず2π縮約の測定結果で判断する方が安全です。

## キャンペーン再起動ループを構造的に止める

今回のようなケースは、通常の「リトライ可能な失敗」ではありません。schedulerに次のような状態を追加するべきです。

```text
BlockedByPrimitiveChange {
    primitive: cpow-periodic-reduction-v2,
    failure_fingerprint: ...,
    dependency_hash: ...
}
```

同じコマンド、同じ角度域、同じ証明生成器ハッシュでタイムアウトした場合は再投入しません。次のいずれかが変わった場合だけ解除します。

* `cpow` emitterのハッシュ
* 範囲縮約補題のハッシュ
* π球定理のハッシュ
* verifier環境digest
* 明示的な性能パラメータ

例外隔離、リトライ、モンスタースキップは一時障害には有効ですが、構造的な証明項爆発には「依存プリミティブ変更待ち」という別状態が必要です。

## ゲート通過後の実行順序

推奨順は次です。

1. ブロック26の残り
2. ブロック28
3. ディスク列
4. Ξ′
5. leaf claim の昇格・監査
6. 先に作った合成定理へ流し込み
7. Z-A/Z-Bの最終合成
8. 全体auditと再現性スナップショット

ただし、ブロック26の前にディスク・Ξ′の代表点を1つずつ通すことは前述のとおりです。

範囲縮約後もセル検証が遅い場合、次のボトルネックは `cpow` 証明のセル内インライン化です。その場合は、`cpow` 点球を独立したpromoted leaf claimとして共有し、複数セルから参照する構造へ進むべきです。最初からそこまで改修すると変更範囲が大きくなるため、これはv2測定後の第二手です。

## 自律エージェントへの指示文

そのまま渡すなら、次の内容が適切です。

> 現在のmainを計算スナップショットとしてv0.2文書化・タグ付けする。成果をKernelChecked/promoted、KernelChecked/unpromoted、数値実証、未完了に分離し、最終合成前の結果を過大表現しない。
> 次に、残るleaf certificateを仮定として最終目標を導くLean合成定理をsorryなしで完成させ、境界・量化子・除算非零条件・零点定義の一致を監査する。
> その後、既存cpow経路を変更せず、exact 2π周期性と有理周期球による`PeriodicReductionV2`を追加する。355/113はπ近似なので2π中心には710/113を用い、近似誤差は必ず`r + |k|ε`としてLean側で検証する。Complex.cpow自体ではなく分解後の実数sin/cos位相だけを縮約する。
> 誤り注入試験、低t差分試験、最悪cpow点、ブロック26セル1個、ディスク点1個、Ξ′点1個のゲートを通す。ゲート合格後のみ、既存成功成果を保持したままブロック26、28、ディスク列、Ξ′、最終合成へ進む。同一失敗のキャンペーン再起動は禁止し、cpow実装または依存補題のハッシュ変更までBlocked状態とする。

したがって、今回承認すべき方針は、**「v0.2を固定した後、合成定理の型を先に閉じ、`cpow` 範囲縮約v2を実装して、ゲート通過後は第1零点近傍エンドゲームまで自律継続」**です。
