## claude codeが自律的にリーマン予想を研究できるようにする

../unified-field-theory/ で統一場理論の研究を行っています。ディレクトリ構成や出力などを参考にしてください。

最も現実的な構成は、次の役割分担です。

| 層      | 技術                   | 役割                                     |
| ------ | -------------------- | -------------------------------------- |
| 制御層    | **Rust**             | エージェント実行、探索予算、状態遷移、成果物管理、再現実験          |
| 主証明系   | **Lean 4 + mathlib** | 数学的真偽を確定する第一の証明核                       |
| 探索計画   | **SWI-Prolog**       | 補題依存関係、未解決障害、戦略適用条件、次に解くべき命題の選択        |
| 機械学習接続 | **Python sidecar**   | LeanDojo-v2、Pantograph、既存Lean証明モデルとの接続 |
| 厳密数値計算 | **FLINT/Arb**        | 反例探索、定数推定、区間演算、検証証明書の生成                |
| 独立監査   | **Coq／現Rocq**        | 重要な橋渡し定理や数値証明書チェッカーの独立検証               |
| 非信頼研究者 | **複数LLM**            | 仮説生成、形式化、証明候補、反例候補、文献整理                |

核心は、**LLMやRustやPrologに「証明済み」と判断させないこと**です。これらはすべて未信頼とし、LeanまたはRocqのカーネルが通った成果物だけを知識ベースに登録します。

リーマン予想自体は現在も未解決ですが、mathlibにはすでに複素リーマンゼータ関数、解析接続、函数等式、自明零点、(\operatorname{Re}(s)\ge1) における非消滅、さらに `RiemannHypothesis : Prop` という正式な命題まで入っています。つまり「リーマン予想をLeanで記述する」段階は終わっており、課題はその型の項、すなわち証明を構成することです。 ([Leanコミュニティ][1])

```lean
import Mathlib.NumberTheory.LSeries.RiemannZeta

#check riemannZeta
#check RiemannHypothesis
```

## 1. システム全体を「証明を伴う研究機関」として設計する

構成は次のようになります。

```text
論文・mathlib・既存証明・数値実験
              │
              ▼
     Mathematical Claim IR
   ┌──────────┼───────────┐
   │          │           │
文献Agent   形式化Agent   反例Agent
   │          │           │
   └──────────┼───────────┘
              ▼
      型付き Blueprint DAG
              │
      Prologによる優先順位付け
              │
   ┌──────────┼──────────────┐
   │          │              │
Lean証明探索  FLINT/Arb     批判Agent
   │          │              │
   │       証明書・反例       │
   └──────────┼──────────────┘
              ▼
       clean-room Lean検証
              │
      内容アドレス成果物DB
              │
       重要結果のみRocq監査
```

単一のLLMに「リーマン予想を証明せよ」と長時間考えさせる方式は避けるべきです。代わりに、まず**すべてのノードがLeanで型検査できる補題DAG**を作り、各ノードを並列に証明し、失敗・反例・不足補題に応じて全体のDAGを書き換えます。

2026年のGoedel-Architectも、自然言語の長い証明計画ではなく、形式的に型付けされた補題DAGを生成し、各ノードを証明して失敗から全体設計を修正する方式を採用しています。再帰的な部分問題分解だけでなく、**大域的な依存関係を維持した設計変更**が重要です。 ([arXiv][2])

## 2. SAILに対応する「Mathematical Research IR」を作る

このプロジェクトの最重要部分は、実は証明探索器ではなく**意味表現層**です。

ソースコードに対するSAILと同様に、数学について次の中間表現を作るべきです。

```rust
pub struct ClaimIr {
    pub binders: Vec<Binder>,
    pub assumptions: Vec<LogicalExpr>,
    pub conclusion: LogicalExpr,

    // 完全修飾されたLean/mathlibシンボル
    pub resolved_symbols: BTreeSet<QualifiedName>,

    // この命題で用いている定義の意味とバージョン
    pub definitions: BTreeMap<QualifiedName, DefinitionRef>,

    // 証明上の依存関係
    pub dependencies: BTreeSet<ClaimId>,

    // prove / refute / establish_equivalence / find_bound
    pub intent: ResearchIntent,

    // 論文、定理番号、既存ライブラリ、生成エージェント
    pub provenance: Vec<EvidenceRef>,

    // 自然言語で意図した数学的意味
    pub semantic_contract: SemanticContract,
}
```

ただし、**IR自身を真理の基準にしてはいけません**。真の基準は、特定のLean環境でelaborateされた式です。IRは検索、比較、文献対応付け、Rocqへの移植に使います。

命題の同一性も一種類ではありません。

1. 構文的に同じ
2. (\alpha) 変換や記法展開を除いて同じ
3. Leanの定義的等価性で同じ
4. 命題として同値だが定義的には異なる
5. 数学者の意図として同じに見えるが、実際には量化範囲などが違う

したがって「正規化した文字列のハッシュ」だけで重複排除してはいけません。次の三層にします。

```text
source_fingerprint
    ↓
elaborated_expression_fingerprint
    ↓
Leanで証明された Eq / Iff エッジ
```

特にリーマン予想では、零点の定義、極 (s=1) の扱い、自明零点の除外、完成ゼータ関数と通常のゼータ関数の対応などがずれると、**正しいが目的とは異なる定理**を証明してしまいます。mathlibの `RiemannHypothesis` も、自明零点と (s=1) を明示的に除外する形で定義されています。 ([Leanコミュニティ][1])

これは証明カーネルでも防げません。カーネルが保証するのは「書かれた命題から結論が導かれたこと」であって、「書かれた命題が人間の意図したリーマン予想であること」ではありません。

## 3. Rustでは状態を型で分離する

`proved: bool` のような状態管理は避けます。証明状態を型パラメータにし、不正な状態遷移を表現不能にします。

```rust
use std::collections::{BTreeMap, BTreeSet};

#[derive(Clone, Copy, Debug, Eq, PartialEq, Ord, PartialOrd, Hash)]
pub struct Digest(pub [u8; 32]);

#[derive(Clone, Copy, Debug, Eq, PartialEq, Ord, PartialOrd, Hash)]
pub struct ClaimId(pub Digest);

pub struct Proposed;

pub struct Elaborated {
    pub lean_environment: Digest,
    pub elaborated_statement: Digest,
}

pub struct KernelChecked {
    pub lean_environment: Digest,
    pub proof_artifact: Digest,
    pub dependencies: BTreeSet<ClaimId>,
    pub reported_axioms: BTreeSet<String>,
}

pub struct IndependentlyAudited {
    pub lean_proof: KernelChecked,
    pub rocq_artifact: Digest,
}

pub struct Claim<S> {
    pub id: ClaimId,
    pub statement: ClaimIr,
    pub state: S,
}

pub enum Rejection {
    IllTyped,
    StatementMismatch,
    SearchExhausted,
    CounterexampleFound { certificate: Digest },
    ForbiddenAxiom { name: String },
    UndeclaredDependency { claim: ClaimId },
    ReproducibilityFailure,
}
```

重要なのは、`KernelChecked` のコンストラクタをprivateにし、固定されたLean検証プロセスだけが生成できるようにすることです。

```rust
pub fn verify_lean_artifact(
    claim: Claim<Elaborated>,
    artifact: UntrustedLeanArtifact,
    verifier: &PinnedLeanVerifier,
) -> Result<Claim<KernelChecked>, Rejection>;
```

データベース上の `"status": "proved"` は権威ではなくキャッシュです。いつでも証明成果物から再計算できるようにします。

成果物は次を含むMerkle DAGにします。

```text
命題
Lean環境のハッシュ
lean-toolchain
mathlibコミット
lake-manifest
証明ソース
依存定理
使用axiom一覧
モデル識別子
プロンプト識別子
乱数seed
探索ログ
失敗ログ
数値証明書
```

## 4. Leanへの接続方法

LeanDojo-v2は、Leanリポジトリのトレース、検索用データセット、検索拡張、whole-proof生成、逐次的証明探索をまとめた基盤です。Pantographは、Leanのproof stateやmetavariableをプログラムから操作するmachine-to-machine interfaceを提供しています。 ([leandojo.org][3])

現時点では、これらをRustで再実装するよりも、

```text
Rust orchestrator
       │ JSON-RPC / stdin-stdout
       ▼
Python Lean sidecar
       │
       ▼
Pantograph / LeanDojo-v2
```

という構成が合理的です。

ただし、Pantograph経由で成功しただけでは受理しません。最終的には別プロセス・新規作業ディレクトリで、

```bash
lake env lean GeneratedProof.lean
```

を実行して再検証します。

命題だけを型検査するときは、`sorry` を入れた定理を生成するのではなく、まず命題を `Prop` として定義します。

```lean
def Claim_a17f : Prop :=
  ∀ s : ℂ, SomePreciselyGeneratedStatement s

#check Claim_a17f
```

証明段階で初めて、

```lean
theorem prove_Claim_a17f : Claim_a17f := by
  -- generated proof
```

を生成します。

受理条件は少なくとも次です。

* clean environmentで再ビルドできる
* `sorry`、`admit`、未申告の独自axiomがない
* foundational axiomsは明示的なallowlistに入っている
* 証明項が参照する定数を走査し、申告依存関係と一致する
* モジュール依存関係に循環や隠れた仮定がない
* 証明と命題の環境ハッシュが一致する
* 数値計算を使った場合は、その証明書を再生できる

単なるソースコード検索で `sorry` を探すだけでは不十分です。生成された証明項の定数依存閉包と、Leanが報告するaxiomを検査する必要があります。

## 5. Prologは「数学的真理」ではなく探索計画に使う

Prologに向いているのは、次のようなメタレベルの推論です。

```prolog
:- table depends_transitively/2.

depends_transitively(A, B) :-
    depends(A, B).

depends_transitively(A, C) :-
    depends(A, B),
    depends_transitively(B, C).

cyclic(A) :-
    depends_transitively(A, A).

blocked(Goal, Missing) :-
    depends(Goal, Missing),
    status(Missing, open).

candidate(Goal, Score) :-
    status(Goal, open),
    \+ cyclic(Goal),
    impact(Goal, Impact),
    estimated_cost(Goal, Cost),
    blocker_reduction(Goal, Reduction),
    Score is Impact * (1 + Reduction) / (1 + Cost).

best_candidate(Goal) :-
    setof(Score-G, candidate(G, Score), Ranked),
    last(Ranked, _-Goal).
```

SWI-Prologにはtabling、incremental tabling、answer subsumptionがあり、補題依存グラフや状態更新後の再計算に適しています。RDFストアもあるため、文献・著者・定理・定義・証明戦略の知識グラフにも使えます。 ([SWI-Prolog][4])

ただし、

```prolog
proved(rh).
```

という事実をPrologが自分で生成してはいけません。許されるのは、

```prolog
kernel_checked(rh, LeanArtifactHash).
```

をRustの検証器から受け取ることだけです。

## 6. エージェントを役割分離する

単一モデルより、以下の役割を独立させた方がよいです。

### Architect

全体戦略と型付き補題DAGを作る。補題は証明前にすべてLeanでelaborateされなければならない。

### Librarian

mathlib、論文、既存形式化から定義・類似定理・使えそうな補題を検索する。名称類似ではなく、型と依存関係を重視する。

### Formalizer

自然言語や論文中の定理をClaim IRとLean命題へ変換する。証明は担当しない。

### Prover

whole-proof生成、逐次tactic探索、Aesop系探索、第一階部分の自動化、既存補題検索を組み合わせる。

### Refuter

有限例、極限ケース、符号、定数、境界条件を探し、間違った補題を早期に破棄する。

### Critic

量化子の欠落、循環論法、必要以上に強い仮定、空虚に真な命題、既知定理の言い換えを検出する。

### Auditor

clean-room検証、依存関係検査、axiom検査、再現性検証を行う。他のエージェントの自然言語説明は信用しない。

さらに、同じ証明モデルを多数決させても信頼性はあまり上がりません。モデル間の合意ではなく、**異なる攻撃面を持つ役割分離**が必要です。

## 7. 数値計算は証明器ではなく反例器・証明書生成器にする

有限個の零点を検査してもリーマン予想全体は証明できません。しかし数値計算には次の大きな価値があります。

* 間違った中間補題を高速に棄却する
* 明示定数の妥当な範囲を推定する
* 補題の仮定不足を発見する
* 解析的証明で必要な不等式を区間演算で確定する
* 有界範囲について形式的定理を作る

Arbは厳密な実・複素ball arithmeticを提供し、現在はFLINTに統合されています。FLINTにはリーマンゼータ零点を扱う機能もあります。 ([arblib.org][5])

ただし、次の二つは区別します。

```text
FLINTがtrueを返した
```

と

```text
FLINTが証明書を生成し、小さなLeanチェッカーがその証明書を検証した
```

後者だけを数学的成果として受理します。

最初は巨大な零点計数証明書ではなく、次のような小さい証明書言語から始めるとよいです。

```text
IntervalAdd
IntervalMul
IntervalInv
PolynomialBound
ExpBound
LogBound
GammaBound
ComplexRectangleBound
SignCertificate
```

LeanとRocqの両方で同じ証明書チェッカーを実装すると、数値部分についてかなり強い独立検証になります。

## 8. Coqは全体の第二実装にしない

Coqは現在Rocqという名称になっています。MathComp Analysisには実解析の形式化資産があります。 ([Rocq][6])

しかし、リーマンゼータ関数周辺の全体系をLeanとRocqで同時開発すると、研究の大半がライブラリ移植になります。したがってRocqは次に限定すべきです。

1. 数値証明書チェッカー
2. 重要な有限的・代数的補題
3. Lean側の定義と数学的意味を結ぶ橋渡し定理
4. 最終結果の核心部分
5. Leanと同じ間違いを共有しにくい独立定義

Lean証明をLLMでRocq構文へ翻訳しただけでは独立検証になりません。両方が同じ誤訳された命題を証明する可能性があるからです。

独立性を得るには、

* 共通の中立仕様から両方の命題を生成する
* またはそれぞれ独立に定義する
* その後、両定義の意味対応を別途監査する

必要があります。

## 9. 最初の研究課題

いきなりリーマン予想全体をルートノードにすると、エージェントはほぼ無限に曖昧な補題を生成します。次の順番がよいです。

### 第0課題: 既存定理再生ベンチマーク

mathlibのゼータ関数周辺から既存定理を抽出し、証明を隠して再証明させます。

評価対象は、

* 命題elaboration成功率
* 必要補題のretrieval
* proof-state探索
* clean-room再検証
* 失敗分類
* 計算予算当たりの証明数

です。

事前学習データに既存証明が含まれている可能性があるため、これだけで研究能力を評価してはいけません。定理の変形、前提マスキング、新規補助補題、時刻で分離した自前リポジトリも評価に含めます。

### 第1課題: リーマン予想の複数定式化の同値性

古いLean形式化にはDirichlet eta関数を用いたリーマン予想の定式化があります。2025年のゼータ・L関数形式化報告では、旧eta形式と現在のゼータ形式の同値性がまだ形式化されていないと説明されていました。まず現在のmathlibを自動検査し、まだ未収録なら、この同値性を最初の本格課題にするのが適切です。 ([GitHub][7])

概念的には、

```lean
theorem riemannHypothesis_iff_etaForm :
    RiemannHypothesis ↔ EtaRiemannHypothesis := by
  ...
```

を目標にします。

これは、

* 異なる定義の意味整合
* 解析接続
* 零点の対応
* 除外点の扱い
* 既存ライブラリ検索
* 有用な中間補題生成

を一度に試せるため、システム評価として優れています。

### 第2課題: 完成ゼータ関数・xi関数との対応

通常のゼータ関数、完成ゼータ関数、xi関数について、

* 零点集合の対応
* 臨界線への変換
* 複素共役対称性
* (s\mapsto1-s) 対称性
* 各形式のRH同値性

をひとつの形式的APIとして整理します。

既存定理があれば再利用し、なければ補います。「新しい数学」より先に、AIが探索しやすいインターフェースを作ることが重要です。

### 第3課題: 厳密数値証明書

有界領域の零点、明示的不等式、特殊関数評価について、FLINT/Arbから証明書を出しLeanで確認します。

### 第4課題: 既知の同値条件の形式化

Liの判定条件、Nyman–Beurling型条件、明示的素数分布評価などから、現在のライブラリ基盤に適合するものを選びます。

ここで重要なのは「同値条件を増やせば自動的にRHに近づく」わけではないことです。同値条件は、**AIが異なる数学領域から探索できる入口を増やす**目的で形式化します。

### 第5課題: 新規中間補題の自律生成

ここから初めて、既存論文に明示されていない補題を生成させます。

良い候補は、

* 既知定理より少し一般化された補題
* 二つの既知補題を統合する抽象化
* 明示定数を改善する不等式
* 数値実験で強く支持され、Leanで明確に記述できる命題
* 複数の未解決ノードを同時に解消する高中心性補題

です。

## 10. Blueprint DAGの更新規則

各ノードは次の状態を持ちます。

```text
Proposed
IllTyped
Elaborated
Blocked
SearchExhausted
NumericallyRefuted
FormallyRefuted
ConditionallyProved
KernelChecked
IndependentlyAudited
Superseded
```

`Unproved` 一種類にまとめてはいけません。

例えば探索が失敗した場合も、

```text
命題が偽
必要な補題がない
適切な既存補題を検索できなかった
時間切れ
形式化が悪い
現在の戦略に向かない
```

は全く別です。

Blueprint更新はおおむね次の純粋関数として扱えます。

```rust
pub fn refine_blueprint(
    graph: Blueprint,
    observations: Vec<Observation>,
    policy: &RefinementPolicy,
) -> Blueprint;
```

更新例は次の通りです。

```text
型エラー
  → 命題を修正する。証明探索しない

小さい反例
  → 命題を破棄、仮定追加候補を別ノードとして生成

証明探索失敗
  → 証明計画を分解、補助補題を生成

複数ノードが同じ障害で停止
  → 共通補題を高優先度化

証明できたが巨大・不安定
  → 抽象化補題を生成して再構成

既知定理と同値
  → 新規成果ではなくequivalence edgeとして保存
```

## 11. 報酬設計

「goalを閉じたら+1」だけでは、エージェントは簡単な定理、空虚な仮定、既知定理の言い換えを大量生産します。

報酬は次のようにします。

```text
+ kernel-checked
+ 複数の未解決ノードを解消
+ 再利用可能な一般補題
+ 厳密な反例を発見
+ 必要仮定を弱めた
+ 証明依存を減らした
+ 既存APIを自然に改善

- 独自axiom
- 循環依存
- 空虚な仮定
- 既知定理の重複
- 数値的根拠だけで一般命題を主張
- 非常に脆い巨大証明
- importした全ライブラリから無制限に補題を使用
```

優先度の一例は、

[
\text{priority}
===============

\frac{
\text{dependency reduction}
\cdot
\text{mathematical impact}
\cdot
\text{expected checkability}
\cdot
\text{novelty}
}{
1+\text{estimated cost}+\text{formalization risk}
}

## 14. 最重要の設計判断

この研究で最も大きなボトルネックは、tactic探索ではありません。

**「何を補題として立てればよいか」を発見する能力**です。

現在の形式証明AIは、明確に与えられた命題を閉じる能力は急速に向上しています。LeanDojo-v2やPantographはその探索基盤を提供していますが、RH級の研究では、命題そのものの発明、意味の整合、長距離依存関係、誤った研究方向の棄却が支配的になります。 ([leandojo.org][3])

したがって、開発資源の配分はおおむね、

```text
35% Claim IR・意味整合・知識グラフ
25% Blueprint生成・更新
20% Lean証明探索
10% 反例・厳密数値計算
10% 独立監査・再現性
```

程度が妥当です。証明モデルの微調整だけに大半を投じるべきではありません。

最初の具体的目標は、**「リーマン予想を証明するエージェント」ではなく、「未信頼のAIが提案した新しい補題・反例・同値性を、誤受理なしで累積できる自律研究システム」**に置くべきです。この基盤が成立すれば、同じシステムを一般化リーマン予想、L関数、素数定理の誤差項、零点密度評価などへ段階的に広げられます。
